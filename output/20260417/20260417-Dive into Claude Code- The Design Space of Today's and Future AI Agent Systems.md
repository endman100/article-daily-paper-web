Dive into Claude Code: The Design Space of Today's and Future AI Agent Systems

深入 Claude Code：當代與未來 AI Agent 系統的設計空間

有一篇論文直接把 Claude Code 的 TypeScript 原始碼拆開來看

不是猜測，不是行銷說詞，是逐行分析後整理出的架構報告

Claude Code 能執行終端機指令、編輯檔案、替使用者呼叫外部服務

研究者從中歸納出五個驅動整個系統設計的核心人類價值觀

這五個價值觀往下延伸為十三條設計原則

每條原則再對應到具體的程式碼決策

這種由價值觀到實作的分析框架，讓我們看到 AI Agent 不只是技術問題

整個設計空間到底長什麼樣子，繼續往下看

👇

----
系統的核心非常簡單，就是一個 while 迴圈

呼叫模型、執行工具、重複，三個步驟一直跑下去

這種簡潔不是偷懶，而是刻意把複雜度推到外圍

真正的工程量集中在圍繞這個迴圈的支撐系統上

有七種模式的權限系統，搭配一個機器學習分類器

有五層壓縮管線處理上下文管理

有四種擴充機制，還有子 Agent 委派與工作階段儲存

這些系統的總程式碼量遠超過那個三步驟的迴圈本身

----
五個核心價值觀是：人類決策權威、安全與保全、穩定執行、能力放大、情境適應

人類決策權威排在第一位，這個順序不是偶然的

設計者明確拒絕把所有控制權交給 AI 自行決定

七種權限模式覆蓋了從全自動執行到每步都需確認的整個範圍

ML 分類器即時評估每個動作的風險等級，不依賴靜態規則

安全與穩定執行是兩個獨立的維度，不能混為一談

安全是關於不做有害的事，穩定執行是關於把正確的事做完

這兩個目標有時相互拉扯，系統設計必須明確選邊

----
上下文視窗的限制是 LLM Agent 最難處理的工程挑戰之一

長時間執行的任務會不斷累積對話與工具回傳，遲早超出模型的處理上限

Claude Code 用了五層壓縮管線來應對這個問題

每一層採取不同策略縮減上下文體積

重要的工具輸出、最近的對話記錄、關鍵錯誤訊息各有不同的保留優先順序

這不是粗暴截斷，而是有策略的摘要與選擇性保留

工作階段的儲存採用只附加不修改的模式

完整的執行歷史因此得以保存，錯誤追蹤與稽核都變得更直接

----
四種擴充機制讓 Claude Code 可以接上外部系統與工作流程

MCP 協定標準化了模型與外部工具之間的溝通介面

插件讓開發者打包整組新工具加入系統

技能是預定義的任務流程，可以被直接呼叫；鉤子則在特定事件觸發時執行自訂邏輯

子 Agent 委派機制讓主 Agent 把複雜任務分拆出去

每個子 Agent 在隔離的 worktree 裡運行，彼此不會互相汙染

這種隔離設計在並行處理多個子任務時格外重要

能力放大這個價值觀，在這四種機制裡得到了最具體的工程表達

----
研究者把 Claude Code 和 OpenClaw 放在一起做對比分析

OpenClaw 是一個開源的多頻道個人助理閘道器，設計脈絡完全不同

兩個系統面對的是同一批反覆出現的設計問題

但部署情境的差異讓答案走向截然不同的方向

Claude Code 對每個動作個別做安全分類，OpenClaw 在周邊層做存取控制

Claude Code 是獨立的 CLI 迴圈，OpenClaw 是嵌入閘道控制平面的執行環境

擴充機制方面，前者在上下文視窗裡延伸能力，後者在全閘道範圍登錄能力

這組對比的意義不在於哪個更好，而在於說明情境決定架構

論文最後識別出六個 AI Agent 系統的開放設計方向

這些方向根植於近期的實證研究、架構分析與政策文獻

如何在長時間任務中維持穩定性，是目前最迫切的工程問題之一

多 Agent 協作的結構化方式仍然缺乏成熟的解法

信任邊界的設計還非常原始，當 Agent 呼叫另一個 Agent，責任歸屬沒有標準答案

情境適應這個價值觀最難被量化，也最難在系統層面被正式定義

從一個 while 迴圈出發，整個設計空間原來這麼廣
