Do AI Coding Agents Log Like Humans? An Empirical Study

《AI 程式碼代理的日誌行為與人類相同嗎？一項實證研究》

這篇研究聚焦在 AI coding agent 寫程式時，是否也像人類開發者一樣替程式碼加上適當的日誌（log）

結論揭示了一個令人意外的事實：AI agent 在日誌撰寫上與人類有顯著落差，而且人類往往在事後默默補救

AI agent 不只寫出功能正確的程式碼，但在維運可見性這塊，它們其實留下了一個巨大的黑洞

👇

----
軟體日誌（logging）是系統維護的基石，它記錄了程式在執行時的狀態與錯誤，讓工程師能在出問題時快速定位根因

傳統上，人類開發者會根據情境判斷應該在哪裡、記錄什麼層級的訊息，這是一種隱性的工程知識

現在 AI coding agent 能直接提交 pull request，理論上也應該一併處理這類非功能性需求

研究團隊為了釐清這個問題，蒐集了 81 個成熟開源儲存庫中共 4,550 個 AI 代理提交的 PR，與 3,276 個人類 PR 進行比對

這是目前規模最大、針對 AI agent 日誌行為的實證研究

----
第一個關鍵發現是：AI agent 修改日誌的頻率比人類低很多

在研究的儲存庫中，有超過一半的案例顯示 AI agent 更動日誌的次數遠低於人類開發者

這意謂著 AI agent 更傾向跳過日誌這個步驟，直接完成功能實作就算完工

然而有趣的是，在那些 AI agent 確實有加入日誌的儲存庫中，它反而會過度記錄

每 1000 行程式碼中，AI agent 加入的日誌數量比人類多出約三成

這種「要不就不動、要動就過頭」的模式，顯示 AI agent 對日誌的判斷缺乏細膩度

----
在日誌層級的使用上，AI agent 能成功模仿人類在錯誤處理時使用 ERROR 層級的習慣

但遇到 INFO 層級這類帶有情境說明性質的日誌時，AI agent 的表現就變得不穩定

INFO 日誌通常是用來記錄「系統目前做到哪一步」的訊息，對日常監控至關重要

AI agent 在這部分的一致性明顯不如人類，說明它對日誌的用途理解是片段的，而非整體性的

這也意謂著 AI agent 產出的程式碼，在可觀測性（observability）這個面向上有結構性缺口

----
研究的第二個方向是分析人類開發者是否有透過自然語言指令來引導 AI agent 的日誌行為

結果顯示，在所有分析的 PR 中，只有不到 5% 的議題描述或指令檔案中包含任何關於日誌的明確要求

更令人擔憂的是，即便有明確的日誌指令，AI agent 也有近七成的機率完全不理會這些要求

無論指令寫得多具體、多詳細，合規率依然偏低，顯示問題不在於指令的清晰度

自然語言指令在控制 AI agent 的日誌行為上，效果非常有限

研究者將此定義為「雙重失效」：指令稀缺，加上代理不服從

----
第三個發現更是揭露了一個隱藏的維護代價

在 AI agent 提交 PR 之後，有超過七成的日誌修復工作是由人類開發者在後續的 commit 中悄悄完成的

這些人類開發者被研究者稱為「沉默的清潔工（silent janitors）」

他們並未在 code review 中公開要求修正，而是直接在後續提交中補上遺漏的日誌或修正不當的記錄

這種隱性的修復負擔，在傳統的開發指標中完全看不見，卻是真實存在的人力成本

當 AI agent 的使用量持續增加，這個沉默負擔也會同步擴大

----
這些發現指向一個核心問題：用自然語言來規範 AI agent 的日誌行為，並不可靠

現有的 AI coding agent 主要被訓練在功能正確性上，通過測試就算完成任務

日誌這類非功能性需求，因為缺乏明確的驗證標準，往往在 agent 的優先序中被排到最後

研究者因此建議，應該考慮引入確定性的防護機制（deterministic guardrails）

例如靜態分析規則、PR 檢查機制，或是自動化的日誌覆蓋率驗證，讓日誌品質有明確的門檻可供檢查

這些方法不依賴 agent 的自覺，而是從工程流程層面強制執行

----
這項研究對 AI 輔助開發的實務有直接啟示

隨著越來越多的團隊採用 AI coding agent 來提升開發速度，程式碼的可維護性與可觀測性可能正在被犧牲

功能正確不等於維運就緒，一個沒有足夠日誌的系統，在生產環境出問題時幾乎等同於黑盒子

工程團隊需要重新審視對 AI agent 的期望範圍，並建立對應的品質驗證流程

單純依賴 AI agent 自主判斷日誌需求，目前看來並不安全

----
這篇研究提醒了我們，AI agent 的能力邊界比表面看起來更複雜

它們能寫出能跑的程式，但「能跑」和「能被維護、能被監控」之間，還有一段距離

日誌只是非功能性需求中的一個例子，安全性、錯誤處理、效能考量，同樣可能面臨類似的代理不足問題

如何讓 AI agent 真正理解並執行工程規範，而不只是通過測試，是接下來整個領域需要面對的挑戰

在那之前，人類開發者仍然是那個在幕後默默補洞的人
